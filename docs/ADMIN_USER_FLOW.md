# Crewith 관리자 웹 사용 흐름

## 작성 기준

- 구현 전 설계 기준으로 작성한다.
- 각 플로우는 목적, 사용자, 진입 경로, 실행 단계, 성공 결과, 예외/오류 상황으로 구성한다.
- "현재 구현됨" 또는 "추후 구현" 표현은 명확히 구분한다.

## 1. 관리자 로그인

**목적**: 운영진이 관리자 웹에 인증하여 모임 운영 기능에 접근한다.

**사용자**: 모임 owner 또는 operator

**진입 경로**: 관리자 웹 URL 직접 접근 → 비인증 상태이면 로그인 화면으로 리다이렉트

**실행 단계**
1. 로그인 화면에서 휴대폰 번호 입력
2. OTP 요청 버튼 클릭
3. SMS로 수신한 OTP 입력 (개발 환경: `123456`)
4. 확인 버튼 클릭
5. 인증 성공 후 모임 선택 또는 대시보드로 이동

**성공 결과**
- JWT 발급 및 브라우저 스토리지에 저장
- 마지막 선택 모임이 있으면 해당 모임 대시보드로 이동
- 가입된 모임이 없으면 모임 생성 화면으로 이동

**예외/오류 상황**
- 잘못된 OTP 입력: 오류 메시지 표시, 재입력 요청
- OTP 만료: 재발송 버튼 노출
- 운영진 권한 없는 계정: 접근 불가 안내
- 네트워크 오류: 재시도 안내

**관련 QA 항목**
- API 3.2 운영진 권한 보호
- 릴리즈 게이트 7.4 라우트 HTTP 200 확인

## 2. 대시보드 확인

**목적**: 운영진이 모임 운영 상태를 한 화면에서 파악한다.

**사용자**: operator 이상

**진입 경로**: 사이드바 > 대시보드 `/`

**실행 단계**
1. 대시보드 화면 진입
2. 상단 지표 카드 확인 (전체 회원, 활성 회원, 미납자, 공지 확인률, 실제 출석률, 회비 수납률)
3. 작업 큐 패널 확인 (가입 신청 대기, 회비 미납자, 다가오는 일정)
4. 각 항목 클릭 시 해당 관리 화면으로 이동 확인

**성공 결과**
- 지표 카드에 실시간 집계 값 표시
- 작업 큐에서 즉각 처리가 필요한 항목 확인 가능
- 무료 체험 상태 배너 표시 (MVP 기간 중)

**예외/오류 상황**
- API 응답 지연: 스켈레톤 로딩 표시
- 데이터 없음 (신규 모임): 0 또는 "-" 표시
- 권한 오류: 로그인 화면 리다이렉트

**관련 QA 항목**
- 관리자 웹 1.1 대시보드 지표 렌더링
- API 3.4 회비 지표 계산

## 3. 회원 등록

**목적**: 운영진이 새 회원을 수동으로 등록하거나 Excel 업로드로 일괄 등록한다.

**사용자**: operator 이상

**진입 경로**: 사이드바 > 구성원 `/members` > 회원 추가 버튼

**실행 단계**
1. 회원 추가 버튼 클릭
2. 이름, 휴대폰 번호, 역할 입력
3. 선택 사항: 생년월일, 성별, 등급 입력
4. 저장 클릭
5. (일괄 등록) Excel 업로드 탭 선택 → 파일 선택 → 업로드

**성공 결과**
- 신규 회원이 목록 최상단에 추가됨
- 역할이 선택 값으로 설정됨, 상태가 `active`로 초기화됨
- Excel 업로드 시 성공/실패 행 수가 결과로 표시됨

**예외/오류 상황**
- 중복 휴대폰 번호: 오류 메시지 표시, 저장 불가
- 필수 필드 누락: 인라인 유효성 오류 표시
- Excel 형식 오류: 행별 오류 목록 표시, 유효한 행은 저장

**관련 QA 항목**
- 관리자 웹 1.3 회원 등록
- 관리자 웹 1.2 사이드바 라우트 이동

## 4. 회원 상태 변경

**목적**: 운영진이 회원의 활동 상태를 변경한다 (휴면, 탈퇴, 강퇴).

**사용자**: operator 이상

**진입 경로**: 구성원 목록 > 회원 행 클릭 > 상세 패널 > 상태 드롭다운

**실행 단계**
1. 회원 목록에서 대상 회원 선택
2. 상세 패널에서 상태 드롭다운 클릭
3. `active` / `dormant` / `left` 중 선택
4. 강퇴 시 삭제 버튼 클릭 → 확인 다이얼로그 → `removed` 처리
5. 변경 확인

**성공 결과**
- 상태가 즉시 변경됨
- `left` / `removed` 처리 시 `personalDataDeleteAt`이 30일 후로 설정됨
- `removed` 회원은 일반 목록에서 숨겨짐

**예외/오류 상황**
- owner를 removed로 변경 시도: 차단, "먼저 owner를 이전해야 합니다" 안내
- 이미 left 상태인 회원의 재활성화: 허용 여부 정책 필요 (가정: 허용)

**관련 QA 항목**
- 관리자 웹 1.5 회원 상태 변경
- 개인정보 5.1 탈퇴/강퇴 후 30일 삭제 예약
- API 3.5 탈퇴/강퇴 회원 숨김

## 5. 역할 변경

**목적**: 운영진 권한을 부여하거나 회수한다.

**사용자**: owner만 가능 (operator 이하 불가)

**진입 경로**: 구성원 목록 > 회원 상세 패널 > 역할 드롭다운

**실행 단계**
1. 회원 상세 패널 진입
2. 역할 드롭다운에서 `owner` / `operator` / `member` 선택
3. owner 역할 부여 시: "본인의 역할이 operator로 변경됩니다" 확인 다이얼로그
4. 변경 확인

**성공 결과**
- 역할이 즉시 변경됨
- owner 이전 시 기존 owner는 operator로 변경됨 (모임당 owner는 1명)
- 변경된 역할에 맞는 권한이 즉시 적용됨

**예외/오류 상황**
- owner가 아닌 계정이 역할 변경 시도: 드롭다운 비활성화 또는 403 처리
- 자기 자신의 owner 역할 자진 해제: 다른 owner가 없으면 차단

**관련 QA 항목**
- 관리자 웹 1.4 회원 역할 변경
- 권한/보안 4.1 모임 간 데이터 격리
- API 3.2 운영진 권한 보호

## 6. 회비 생성

**목적**: 운영진이 정기 회비 또는 일회성 비용 항목을 생성한다.

**사용자**: operator 이상

**진입 경로**: 사이드바 > 회비 `/fees` > 회비 생성 버튼

**실행 단계**
1. 회비 유형 선택 (정기 / 일회성)
2. **정기 회비**: 금액, 납부일, 납부 간격, 유예기간 입력
3. **일회성 비용**: 제목, 금액, 납부 기한 입력
4. 대상 회원 선택 (전체 또는 특정 회원 수동 지정)
5. 저장

**성공 결과**
- 회비 항목이 목록에 추가됨
- 대상 회원별 `unpaid` 상태의 납부 레코드 자동 생성됨
- 정기 회비는 납부 간격 기준으로 향후 항목 자동 생성 (스케줄 작업)

**예외/오류 상황**
- 금액 0 또는 음수 입력: 유효성 오류 표시
- 납부 기한이 과거 날짜: 경고 표시 후 저장 허용 또는 차단 (정책 결정 필요)
- 대상 회원 0명: 저장 불가 안내

**관련 QA 항목**
- 관리자 웹 1.6 회비 항목 생성

## 7. 납부 상태 변경

**목적**: 운영진이 회원의 회비 납부 여부를 기록한다.

**사용자**: operator 이상

**진입 경로**: 회비 화면 > 회비 항목 선택 > 납부 현황 테이블

**실행 단계**
1. 회비 항목 클릭하여 납부 현황 테이블 진입
2. 납부 처리: 회원 행에서 `unpaid` → `paid` 토글 클릭
3. 면제 처리: 드롭다운에서 `exempt` 선택
4. 납부일 직접 수정: 날짜 입력 필드 클릭 후 수정
5. 변경 내역 확인

**성공 결과**
- 납부 상태가 즉시 변경됨
- `paid` 처리 시 `paidAt`이 현재 시각으로 저장됨
- 날짜 직접 수정 시 입력한 날짜로 `paidAt` 저장됨
- 처리한 운영진 정보가 `recordedBy`에 저장됨

**예외/오류 상황**
- 이미 `paid` 상태를 다시 `paid`로 변경: 아무 동작 없음
- `exempt` → `paid` 전환: 가능, 납부 처리로 기록

**관련 QA 항목**
- 관리자 웹 1.7 납부 상태 변경
- API 3.4 회비 지표 계산

## 8. 미납자 확인

**목적**: 운영진이 유예기간이 지난 미납자 목록을 확인하고 알림을 발송한다.

**사용자**: operator 이상

**진입 경로**: 회비 화면 > 미납자 탭 또는 대시보드 작업 큐 > 미납자 항목

**실행 단계**
1. 미납자 목록 확인 (유예기간 경과 기준 필터링)
2. 개인 단위 또는 전체 선택
3. 알림 발송 버튼 클릭
4. 발송 확인 다이얼로그 확인

**성공 결과**
- 미납자 목록이 납부 기한 초과 기준으로 정확히 필터링됨
- 발송 후 알림 로그에 기록됨
- 발송 완료 토스트 메시지 표시

**예외/오류 상황**
- 미납자 없음: "미납자가 없습니다" 빈 상태 표시
- FCM 미연동 환경 (개발): 로그만 기록, 실제 발송 없음 안내

**관련 QA 항목**
- 알림/리마인더 6.1 회비 미납 리마인더 발송
- 관리자 웹 1.15 리마인더 발송

## 9. 일정 생성

**목적**: 운영진이 모임 일정을 등록하고 회원에게 공개한다.

**사용자**: operator 이상

**진입 경로**: 사이드바 > 일정 `/events` > 일정 생성 버튼 또는 대시보드 > 일정 생성 버튼

**실행 단계**
1. 일정 생성 폼 진입
2. 제목, 시작 일시, 종료 일시 입력
3. 장소명, 주소 입력 (네이버 지도 연동은 추후 구현)
4. 참석 의사 응답 마감일 선택
5. 공개 대상 선택 (전체 회원 / 운영진 전용)
6. 저장

**성공 결과**
- 일정이 목록에 추가됨
- 전체 공개 일정은 회원 앱에도 표시됨
- 운영진 전용 일정은 운영진 계정에만 표시됨

**예외/오류 상황**
- 종료 일시가 시작 일시 이전: 유효성 오류
- 응답 마감일이 시작 일시 이후: 경고 표시

**관련 QA 항목**
- 관리자 웹 1.8 일정 생성
- 개인정보 5.4 운영진 전용 데이터 일반 회원 노출 차단

## 10. 참석 의사 확인

**목적**: 운영진이 일정에 대한 회원들의 사전 참석 의사를 확인한다.

**사용자**: operator 이상

**진입 경로**: 일정 목록 > 일정 클릭 > 참석 의사 탭

**실행 단계**
1. 일정 상세 화면에서 참석 의사 탭 선택
2. 참석/불참/미응답 인원 수 확인
3. 각 그룹별 회원 목록 확인
4. 응답 마감 여부 확인

**성공 결과**
- 참석/불참/미응답 수가 정확히 집계됨
- 응답 마감 후에는 수정 불가 표시
- 미응답자에게 리마인더 발송 버튼 제공 (선택)

**예외/오류 상황**
- 응답 마감일 미설정: 응답 마감 관련 UI 비표시
- 회원이 없는 일정: 빈 목록 표시

**관련 QA 항목**
- 관리자 웹 1.9 참석 의사 응답 확인
- 알림/리마인더 6.3 일정 참석 응답 리마인더

## 11. 실제 출석 체크

**목적**: 운영진이 일정 당일 실제 출석 여부를 기록한다.

**사용자**: operator 이상

**진입 경로**: 일정 상세 > 출석부 탭

**실행 단계**
1. 일정 상세에서 출석부 탭 선택
2. 회원별 출석 상태 선택 (출석 / 지각 / 결석)
3. 동반 인원 수 입력 (선택)
4. 저장
5. 출석률 확인

**성공 결과**
- 각 회원의 출석 상태가 저장됨
- 동반 인원 수가 저장됨
- 실제 출석률과 참석 의사 대비 출석률이 계산됨
- 처리한 운영진 정보가 기록됨

**예외/오류 상황**
- 이미 체크된 출석 수정: 덮어쓰기 허용
- 대상 회원 없음: 빈 출석부 표시

**관련 QA 항목**
- 관리자 웹 1.10 출석부 작성
- 권한/보안 4.1 모임 간 데이터 격리

## 12. 공지 작성

**목적**: 운영진이 모임 공지를 작성하여 회원에게 전달한다.

**사용자**: operator 이상

**진입 경로**: 사이드바 > 공지 `/notices` > 공지 작성 버튼 또는 대시보드 > 공지 작성 버튼

**실행 단계**
1. 공지 작성 폼 진입
2. 제목, 내용 입력
3. 공개 대상 선택 (전체 회원 / 운영진 전용)
4. 저장 또는 예약 발송 (추후 구현)
5. 저장 후 앱 푸시 발송 여부 확인

**성공 결과**
- 공지가 목록 최상단에 추가됨
- 전체 공개 공지는 회원 앱에도 표시됨
- 앱 푸시 발송됨 (FCM 연동 시)

**예외/오류 상황**
- 제목 또는 내용 비어 있음: 저장 불가 안내
- FCM 미연동: 공지는 저장되나 푸시 미발송, 안내 표시

**관련 QA 항목**
- 관리자 웹 1.11 공지 작성
- 알림/리마인더 6.4 앱 알림 목록 표시

## 13. 공지 확인률 확인

**목적**: 운영진이 특정 공지의 확인 현황을 파악하고 미확인자에게 리마인더를 발송한다.

**사용자**: operator 이상

**진입 경로**: 공지 목록 > 공지 클릭 > 확인 현황 탭

**실행 단계**
1. 공지 상세 화면에서 확인 현황 탭 선택
2. 확인/미확인 회원 목록 확인
3. 확인률(%) 확인
4. 미확인자 선택 후 리마인더 발송

**성공 결과**
- 확인/미확인 회원이 정확히 구분됨
- 확인률이 공개 대상 회원 수 기준으로 계산됨
- 리마인더 발송 후 알림 로그 기록됨

**예외/오류 상황**
- 모든 회원이 확인: "전원 확인 완료" 표시, 발송 버튼 비활성화
- 운영진 전용 공지의 확인률: 운영진 범위로 계산

**관련 QA 항목**
- 관리자 웹 1.12 공지 확인률 조회
- 알림/리마인더 6.2 공지 미확인 리마인더 발송

## 14. 가입 신청 승인/거절

**목적**: 운영진이 공개 모임 가입 신청을 검토하고 처리한다.

**사용자**: operator 이상

**진입 경로**: 사이드바 > 가입/초대 `/join` > 가입 신청 탭 또는 대시보드 작업 큐

**실행 단계**
1. 가입 신청 목록 확인 (신청자 이름, 연락처, 가입 인사 포함)
2. 신청 항목 클릭하여 상세 확인
3. 승인 또는 거절 버튼 클릭
4. 처리 결과 확인

**성공 결과**
- 승인: 신청자가 `active` 상태 `member`로 등록됨, 신청 앱 푸시 발송
- 거절: 신청 상태가 `rejected`로 변경됨, 거절 알림 발송 (선택)
- 처리 운영진과 처리 시각이 기록됨

**예외/오류 상황**
- 이미 처리된 신청 재처리 시도: 처리 불가 안내
- 신청자가 이미 다른 경로로 가입된 경우: 중복 멤버십 방지 안내

**관련 QA 항목**
- 관리자 웹 1.13 가입 신청 승인/거절
- 알림/리마인더 6.4 앱 알림 목록 표시
- 권한/보안 4.3 타 회원 정보 직접 접근 차단

## 15. 초대 링크 생성

**목적**: 운영진이 비공개 모임 가입용 초대 링크를 생성하여 공유한다.

**사용자**: operator 이상

**진입 경로**: 가입/초대 화면 > 초대 링크 탭 > 링크 생성 버튼

**실행 단계**
1. 링크 생성 버튼 클릭
2. 유효 기간 선택 (7일 / 30일 / 90일)
3. 생성 확인
4. 생성된 링크 URL 복사
5. 필요 시 비활성화 버튼으로 링크 무효화

**성공 결과**
- 초대 링크가 목록에 추가됨
- 만료일이 선택 기간 기준으로 계산됨
- 링크 URL이 클립보드에 복사됨
- 비활성화 후 해당 링크로 가입 불가

**예외/오류 상황**
- 이미 비활성화된 링크 재활성화 시도: 차단 또는 새 링크 생성 유도
- 클립보드 복사 실패: 링크 URL 텍스트 직접 노출

**관련 QA 항목**
- 관리자 웹 1.14 초대 링크 생성
- 권한/보안 4.2 초대 링크 만료/비활성화

## 16. 리마인더 발송

**목적**: 운영진이 미납자, 미확인자, 미응답자에게 일괄 알림을 발송한다.

**사용자**: operator 이상

**진입 경로**: 사이드바 > 리마인더 `/reminders`

**실행 단계**
1. 리마인더 화면에서 발송 유형 선택 (회비 미납 / 공지 미확인 / 일정 미응답)
2. 대상 항목 선택 (회비 항목, 공지, 일정 각각 선택)
3. 발송 대상 미리보기 확인
4. 발송 버튼 클릭
5. 발송 결과 확인

**성공 결과**
- 발송 대상이 조건에 맞게 필터링됨
- FCM 발송 후 알림 로그에 기록됨
- 발송 완료 메시지 표시

**예외/오류 상황**
- 발송 대상 없음: "발송 대상이 없습니다" 표시
- FCM 미연동 (개발): 로그만 기록, 실제 발송 미수행 안내
- 과도한 발송 빈도: 마지막 발송 후 일정 시간 내 재발송 차단 (정책 결정 필요)

**관련 QA 항목**
- 관리자 웹 1.15 리마인더 발송
- 알림/리마인더 6.1 회비 미납 리마인더 발송
- 알림/리마인더 6.2 공지 미확인 리마인더 발송
- 알림/리마인더 6.3 일정 참석 응답 리마인더
