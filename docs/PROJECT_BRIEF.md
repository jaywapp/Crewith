# Crewith 프로젝트 브리프

## 1. 서비스 정의

Crewith는 동호회와 모임에서 구성원, 회비, 일정, 공지사항 등을 쉽게 관리할 수 있도록 돕는 웹/앱 서비스다.

## 2. 주요 사용자

- 동호회 운영진
- 모임 관리자
- 일반 회원

## 3. 해결하려는 문제

현재 많은 동호회와 모임은 운영 정보를 여러 도구에 나누어 관리한다.

- 명부 관리: Excel, Google Sheets
- 일정 관리: 카카오톡 캘린더, 별도 캘린더 앱
- 공지 전달: 카카오톡 채팅방, 게시글, 메시지
- 회비 관리: 계좌 이체 내역, 수기 장부, 스프레드시트

이 방식은 정보가 분산되어 있어 운영진이 반복적으로 데이터를 정리해야 하고, 회원도 최신 정보를 한 곳에서 확인하기 어렵다.

## 4. 핵심 가치

Crewith는 동호회 운영에 필요한 구성원, 회비, 일정, 공지사항 데이터를 하나의 서비스에서 관리하고, 각 기능의 데이터를 연결해 운영 부담을 줄인다.

예를 들어 구성원 정보는 회비 납부 상태, 일정 참석 여부, 공지 확인 여부와 연결될 수 있다.

## 5. 현재 확정된 방향

- 웹과 앱을 모두 고려한다.
- 운영진과 일반 회원을 모두 사용자로 본다.
- MVP는 분산된 운영 도구를 하나로 모으는 데 집중한다.
- 핵심 도메인은 구성원, 회비, 일정, 공지사항이다.
- 초기 타깃은 스포츠 동호회다.
- 초기 기준 모임 규모는 10~30명이다.
- MVP 기능 우선순위는 구성원 명부 관리, 회비 납부 관리, 일정/참석 관리, 공지 전달/확인, 통합 대시보드 순서로 본다.
- 기본 권한은 모임장, 운영진, 일반회원으로 구분한다.
- 사용자는 모임마다 서로 다른 권한을 가질 수 있다.
- 모임은 공개 모임과 비공개 모임을 지원한다.
- 공개 모임은 가입 신청 후 승인 방식, 비공개 모임은 초대 링크 가입 방식을 기본으로 한다.
- 공개 모임 가입 신청 시 사용자는 간단한 인적사항과 가입 인사를 입력한다.
- 가입 신청 승인/거절 결과는 앱 푸시로 알린다.
- 비공개 모임 초대 링크의 만료 기간은 운영진이 설정한다.
- MVP에서 회비는 실제 결제 처리 없이 납부 기록과 미납 관리에 집중한다.
- 회비는 모임별 납부일과 납부 간격을 설정하고, 기준에 따라 납부 상태를 자동 생성한다.
- 회비 미납 유예기간은 모임별로 설정한다.
- 회비 납부 기록은 운영진이 토글 방식으로 입력한다.
- 회비 납부 토글 시 오늘 날짜가 자동 기록되며, 운영진이 날짜를 직접 수정할 수 있다.
- 일정은 사전 참석 의사와 실제 출석부를 구분한다.
- 1차 MVP 출석부는 운영진이 체크하며, 회원 직접 체크인은 추후 확장 여부를 결정한다.
- 참석 의사 응답은 운영진이 정한 마감 시간까지 수정할 수 있다.
- 장소 정보는 지도 연동을 전제로 한다.
- 지도 연동은 네이버 지도를 기준으로 한다.
- 알림 기능은 MVP에 포함한다.
- 사용자는 여러 모임에 가입할 수 있어야 한다.
- 알림 채널은 앱 푸시를 우선한다.
- 미납 알림은 모임별 설정에 따라 자동 발송할 수 있어야 한다.
- 일정 알림은 모임별로 설정할 수 있어야 한다.
- 공지 미확인 회원에게 자동 리마인드를 보낼 수 있어야 한다.
- 공지 확인 여부는 회원이 공지를 열람하면 자동 확인 처리한다.
- 공지 댓글은 모든 회원이 작성할 수 있다.
- 공지는 전체 회원 또는 운영진만 보기 권한을 설정할 수 있어야 한다.
- 일정도 전체 회원 또는 운영진만 보기 권한을 설정할 수 있어야 한다.
- 회비 항목은 전체 회원 또는 특정 대상 회원에게만 부과할 수 있어야 한다.
- 특정 회원 대상 지정은 운영진이 수동으로 선택하는 방식으로 한다.
- 일정 참석자에게 일회성 회비를 자동 부과하는 기능은 MVP에서 제외한다.
- 운영진 전용 공지와 일정은 일반 회원 목록에서도 숨긴다.
- 회원가입/로그인은 휴대폰 인증을 우선한다.
- 수익 모델은 모임당 월 구독이며, 회원 수에 따라 요금이 달라지는 구조를 고려한다.
- 구독 요금제 초안은 30명 이하, 100명 이하, 300명 이하 구간으로 나눈다.
- 모임 생성 후 30일 무료 체험을 제공한다.
- MVP에서는 실제 구독 결제를 구현하지 않는다.
- 초기 출시는 관리자 웹과 회원 앱을 동시에 고려한다.
- 운영진 관리 기능은 모바일 앱 안에서도 제공하고, 별도 웹 관리자 화면도 제공한다.
- 앱은 Flutter 기반으로 개발하되, 초기 검증은 Android를 우선한다.
- 관리자 웹은 Next.js 기반으로 개발한다.
- 백엔드는 NestJS 기반 API 서버를 우선 고려한다.
- 데이터베이스는 관리형 PostgreSQL을 사용한다.
- MVP 인증은 SMS 기반 휴대폰 인증으로 진행하고, 추후 PASS 등 국내 본인인증을 추가한다.
- 앱 푸시는 Firebase Cloud Messaging(FCM)을 사용한다.
- Firebase는 푸시 알림 용도로만 사용한다.
- 관리자 웹은 Vercel에 배포한다.
- 백엔드 API는 Render, Railway, Fly.io 같은 PaaS에 배포한다.
- 파일 저장소는 Cloudflare R2 또는 S3 호환 스토리지를 고려한다.
- 웹 관리자에서는 회원 일괄 등록, 회비 현황 관리, 일정 생성, 공지 작성, 출석부 관리, 통계 확인을 편하게 처리할 수 있어야 한다.
- 기존 Excel 또는 Google Sheets 기반 명부를 가져올 수 있는 엑셀 업로드 기능이 필요하다.
- 엑셀 업로드는 업로드 즉시 등록되는 방식으로 한다.
- MVP 통계는 전체 회원 수, 활성 회원 수, 회비 미납자 수, 일정 참석률, 공지 확인률, 월별 회비 수납률을 포함한다.
- 출석 통계는 실제 출석 기준과 참석 의사 대비 출석률을 모두 제공한다.
- 파일/사진 업로드는 회원 프로필 사진을 대상으로 한다.
- 회원 개인정보 공개 범위는 운영진이 설정할 수 있어야 한다.
- 탈퇴/강퇴/휴면 회원 개인정보는 30일 후 삭제하되, 회비와 참석 이력은 보존한다.
- 공개 모임 검색/탐색 기능은 MVP에서 제외한다.
- 스포츠 종목은 모임 단위로 관리한다.

## 6. 아직 확인이 필요한 영역

- 백엔드/API 상세 설계
- PostgreSQL 데이터 모델
- PaaS 제공자 선택
